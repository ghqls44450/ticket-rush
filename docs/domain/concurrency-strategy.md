# 동시성 제어 전략

좌석 점유 기능은 같은 좌석에 여러 요청이 동시에 들어오는 상황을 전제로 설계한다. 이 문서는 좌석 임시 점유와 예매 확정 흐름에서 비교할 동시성 제어 전략과 검증 기준을 정리한다.

## 문제 시나리오

예매 오픈 시점에는 같은 회차의 같은 좌석에 여러 사용자의 요청이 동시에 들어올 수 있다. 이때 하나의 좌석은 한 명의 사용자에게만 점유되어야 한다.

기본 상태 흐름은 다음과 같다.

```text
AVAILABLE -> HELD -> CONFIRMED
```

동시성 제어의 핵심 질문은 다음과 같다.

- 같은 좌석에 동시에 `HELD` 요청이 들어왔을 때 중복 점유가 발생하지 않는가
- 실패한 요청은 일관된 에러로 응답하는가
- 충돌이 많은 상황에서 응답 시간과 실패율이 어떻게 변하는가

## 1차 적용 전략

첫 구현은 `Pessimistic Lock`을 기준으로 한다.

이유는 다음과 같다.

- 좌석 점유는 중복 허용이 치명적인 흐름이라, 초기 구현에서는 성능보다 정합성을 먼저 확인해야 한다.
- `SELECT ... FOR UPDATE`를 사용하면 같은 좌석 row에 대한 동시 접근을 DB가 순차화하므로 동작을 이해하고 검증하기 쉽다.
- 이미 좌석 상태 전이 규칙이 `AVAILABLE -> HELD`를 기준으로 정의되어 있어, 트랜잭션 안에서 상태 확인과 변경을 함께 처리하기 좋다.
- 이후 Optimistic Lock과 Redis 분산락을 비교할 때 기준선으로 삼기 쉽다.

따라서 다음 구현 단계에서는 먼저 `Pessimistic Lock` 기반 좌석 임시 점유 API를 만들고, 동시 요청 테스트로 중복 점유가 발생하지 않는지 확인한다. Optimistic Lock과 Redis 분산락은 같은 시나리오에서 비교할 후속 전략으로 둔다.

## 비교할 전략

### Optimistic Lock

DB row에 version 값을 두고, 업데이트 시점에 version이 일치할 때만 상태를 변경한다.

적용 흐름:

```text
좌석 조회 -> version 확인 -> status/version 조건으로 UPDATE -> 영향 row 수 확인
```

장점:

- 락을 오래 잡지 않아 충돌이 적은 상황에서 유리하다.
- DB 대기 시간이 상대적으로 적다.
- 구현과 테스트가 비교적 단순하다.

단점:

- 충돌이 많으면 실패와 재시도가 늘어난다.
- 재시도 정책을 별도로 설계해야 한다.

우선 검증할 내용:

- 동일 좌석 동시 점유 요청 시 성공 요청이 1건으로 제한되는지
- 충돌 시 실패 응답이 일관적인지
- 충돌률이 높을 때 재시도 없이도 응답 시간이 안정적인지

### Pessimistic Lock

DB에서 좌석 row를 조회할 때 `SELECT ... FOR UPDATE`로 row lock을 잡고, 트랜잭션 안에서 상태를 변경한다.

적용 흐름:

```text
트랜잭션 시작 -> SELECT FOR UPDATE -> 상태 확인 -> UPDATE -> 커밋
```

장점:

- 하나의 좌석에 대한 동시 접근을 DB가 순차화한다.
- 데이터 정합성을 강하게 보장하기 쉽다.
- 충돌이 많은 상황에서도 중복 점유를 막는 흐름이 명확하다.

단점:

- 대기 시간이 길어질 수 있다.
- 데드락과 타임아웃을 고려해야 한다.
- 트랜잭션 범위가 길어지면 처리량에 영향을 줄 수 있다.

우선 검증할 내용:

- 같은 좌석에 대한 요청이 순차 처리되는지
- lock wait timeout이나 deadlock 발생 여부
- P95/P99 응답 시간이 얼마나 증가하는지

### Redis 분산락

Redis의 `SET NX EX` 방식으로 좌석 단위 락을 잡고, 락을 획득한 요청만 DB 상태를 변경한다.

적용 흐름:

```text
Redis 락 획득 -> DB 상태 확인 및 변경 -> Redis 락 해제
```

장점:

- 여러 애플리케이션 인스턴스에서 같은 좌석 락을 공유할 수 있다.
- DB row lock 시간을 줄일 수 있다.
- 좌석 단위로 세밀하게 락 key를 만들 수 있다.

단점:

- 락 만료 시간과 작업 시간이 어긋나면 정합성 문제가 생길 수 있다.
- 락 해제 실패, Redis 장애, 네트워크 지연을 고려해야 한다.
- DB 트랜잭션과 Redis 락의 책임 범위를 명확히 나눠야 한다.

우선 검증할 내용:

- 동일 좌석 동시 요청에서 락 획득 요청이 1건으로 제한되는지
- 락 만료 시간이 짧거나 긴 경우 실패 패턴이 어떻게 달라지는지
- Redis 장애 시 fallback 또는 실패 응답 기준이 명확한지

## 전략별 비교 기준

| 기준 | Optimistic Lock | Pessimistic Lock | Redis 분산락 |
|---|---|---|---|
| 중복 점유 방지 | version 조건으로 방지 | DB row lock으로 방지 | Redis lock key로 방지 |
| 충돌 많은 상황 | 실패/재시도 증가 | 대기 시간 증가 | 락 획득 실패 증가 |
| 구현 복잡도 | 낮음 | 중간 | 중간 이상 |
| 운영 고려사항 | 재시도 정책 | timeout, deadlock | TTL, 락 해제, Redis 장애 |
| 다중 인스턴스 대응 | DB 중심으로 가능 | DB 중심으로 가능 | Redis 중심으로 가능 |

## 측정 지표

전략 비교 시 다음 지표를 남긴다.

- 총 요청 수
- 성공 수
- 실패 수
- 중복 점유 발생 여부
- 평균 응답 시간
- P95/P99 응답 시간
- lock wait timeout 또는 deadlock 발생 수
- Redis 락 획득 실패 수
