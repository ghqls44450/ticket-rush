# 읽기/쓰기 분리와 stale read 대응 기준

읽기/쓰기 분리 이후 조회 트래픽은 slave로 분산할 수 있지만, 복제 지연이 있으면 쓰기 직후 조회에서 오래된 데이터가 보일 수 있다. 이 문서는 현재 라우팅 규칙을 정리하고, 어떤 조회에서 stale read를 허용할 수 있는지와 어떤 조회를 master로 강제해야 하는지 기준을 남긴다.

## 현재 라우팅 규칙

현재 애플리케이션의 기본 라우팅 규칙은 다음과 같다.

- `@Transactional(readOnly = true)` 트랜잭션은 slave로 라우팅한다.
- 그 외 트랜잭션은 master로 라우팅한다.
- slave 선택 정책은 현재 단일 slave 기준으로 시작하며, 분산 선택과 장애 감지는 후속 이슈에서 확장한다.

이 규칙은 조회 부하를 slave로 분산하는 데는 유리하지만, 모든 조회에 최신 상태를 보장하지는 않는다.

## stale read가 문제되는 이유

stale read는 쓰기 요청이 master에 반영된 직후, 같은 사용자가 이어서 조회한 데이터가 slave에는 아직 반영되지 않아 이전 상태로 보이는 상황을 뜻한다.

대표적인 문제는 다음과 같다.

- 사용자는 성공 응답을 받았는데 직후 화면에는 이전 상태가 보인다.
- 후속 요청이 오래된 데이터를 기준으로 동작해 잘못된 재시도나 중복 행동으로 이어질 수 있다.
- 운영 검증이나 장애 분석 시 최신 상태를 기대했지만 slave 반영 지연 때문에 오해가 생길 수 있다.

즉 stale read는 단순한 조회 지연이 아니라, 사용자 경험과 후속 처리의 기준을 흔들 수 있는 문제다.

## stale read 후보 시나리오

### 1. 좌석 점유 직후 좌석 상태 재조회

좌석 점유 API는 `AVAILABLE -> HELD` 상태 전이를 수행한다. 사용자가 점유 성공 직후 같은 회차의 좌석 상태를 다시 조회하면, slave 반영 전에는 아직 `AVAILABLE`로 보일 수 있다.

영향:

- 사용자는 점유 성공 여부를 신뢰하기 어렵다.
- 같은 좌석을 다시 시도하거나 화면을 반복 새로고침할 수 있다.

### 2. 예매 확정 직후 예약/결제 상태 조회

예매 확정 API는 좌석 상태 변경과 함께 `reservation`, `payment`를 생성한다. 직후 예약 내역이나 결제 상태를 slave에서 조회하면 아직 row가 없거나 이전 상태로 보일 수 있다.

영향:

- 성공 직후에도 예약 내역이 없다고 보일 수 있다.
- 사용자 입장에서는 결제가 반영되지 않은 것처럼 느껴질 수 있다.

### 3. 운영 확인용 직후 조회

운영 확인이나 장애 대응 과정에서는 “지금 막 처리된 결과”를 바로 확인해야 하는 경우가 있다. 이때 stale read가 발생하면 실제 장애와 복제 지연을 구분하기 어려워진다.

영향:

- 장애 원인 파악이 늦어진다.
- 잘못된 수동 조치로 이어질 수 있다.

## Master read가 필요한 기준

다음 조건 중 하나에 해당하면 slave 조회 대신 master 조회를 우선 검토한다.

- 같은 사용자 요청 흐름 안에서 쓰기 직후 이어지는 조회인 경우
- 사용자에게 즉시 최신 상태를 보여줘야 하는 경우
- stale read가 잘못된 후속 행동으로 이어질 수 있는 경우
- 운영 검증, 장애 대응, 정합성 확인처럼 최신성이 우선인 경우

이 기준은 “최신 상태가 아니면 사용자가 잘못 판단할 가능성이 있는가”를 중심으로 판단한다.

## Slave read를 유지할 수 있는 조회

다음 조회는 약간의 복제 지연이 있어도 slave 유지가 가능하다.

- 공연 목록 조회
- 공연 회차 목록 조회
- 일반 좌석 목록 조회
- 통계성, 탐색성, 운영 즉시 판단과 무관한 조회

이런 조회는 최신성보다 조회 부하 분산 효과가 더 중요하다.

## 현재 프로젝트에 우선 적용할 판단

현재 흐름 기준으로 우선 검토할 master read 후보는 다음과 같다.

- 좌석 점유 직후 같은 좌석/회차 상태 확인
- 예매 확정 직후 예약/결제 상태 확인

반대로 다음 조회는 현재 규칙대로 slave read를 유지할 수 있다.

- 공연 목록 조회
- 공연 단건 조회
- 공연 회차 목록 조회
- 일반 좌석 목록 조회

즉 모든 조회를 master로 되돌리지 않고, stale read의 영향이 큰 흐름에만 예외를 좁게 적용하는 것이 기본 방향이다.

## 다음 단계 구현 방향

다음 구현 단계에서는 다음 방향을 우선 검토한다.

- 기본 `readOnly -> SLAVE` 규칙은 유지한다.
- 특정 조회만 명시적으로 master read를 강제할 수 있는 context 또는 annotation 방식을 추가한다.
- master read 강제 범위는 stale read 영향이 큰 유스케이스에 한정한다.

복제 지연 대응은 “모든 조회를 최신으로 만든다”가 아니라, “최신성이 필요한 조회를 식별하고 제한적으로 master를 사용한다”는 방향으로 진행한다.
